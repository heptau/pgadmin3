Первичные ключи

При проектировании таблицы всегда следует помнить о том, как таблица будет адресоваться в дальнейшем. В большинстве случаев вам понадобится идентификатор для уникального обращения к конкретной строке; этот идентификатор должен быть создан как первичный ключ. Первичный ключ не обязательно должен состоять из одного столбца; он может содержать столько столбцов, сколько необходимо для уникальной идентификации строки. Если вам нужно много столбцов (эмпирическое правило: больше 3), возможно, стоит придумать дополнительный столбец с удобным типом данных, например serial или bigserial, который будет содержать первичный ключ.

Только в редких случаях не имеет смысла использовать первичный ключ. Это означает, что отсутствие первичного ключа является довольно сильным индикатором того, что таблица спроектирована не полностью; именно поэтому появится Совет, если вы создадите таблицу без первичного ключа.

Если вы посмотрите на системные таблицы PostgreSQL, вы обнаружите, что ни одна из них не имеет первичного ключа. В чём же дело? На самом деле все эти таблицы имеют один или два столбца (обычно только OID), которые уникально идентифицируют каждую строку, соблюдая второе правило для первичного ключа: не допускать NULL и быть покрытыми индексом для более быстрого доступа. Использование OID имеет исторические причины и не является лучшим выбором для проектирования пользовательских таблиц. PostgreSQL всё ещё использует это для обратной совместимости, и хотя более новый подход, вероятно, использовал бы явные первичные ключи, сейчас это уже не изменится.

Как показывает случай системных таблиц, цель уникальности и быстрого доступа может быть достигнута и другими подходами, помимо первичного ключа. Тем не менее, для чистоты модели данных настоятельно рекомендуется использовать первичные ключи для этой цели.
