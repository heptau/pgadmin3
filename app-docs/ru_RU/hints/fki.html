Покрывающие индексы для внешних ключей

Внешний ключ используется для логической связи двух таблиц и ограничения изменений, которые могут быть к ним применены.

Всё начинается с главной таблицы, которая будет ссылочной таблицей. Она должна иметь первичный ключ (на самом деле ссылочные столбцы должны быть только не NULL и покрыты уникальным индексом, но наличие первичного ключа — очень хорошая рекомендация). Подчинённая таблица зависит от главной таблицы в том смысле, что столбцы, на которых создаётся внешний ключ, должны соответствовать главной таблице для любого вставляемого или обновляемого ключа. Подчинённая таблица является ссылающейся таблицей внешнего ключа.

Внешний ключ ограничивает не только ссылающуюся таблицу, но и ссылочную; тип ограничения может быть RESTRICT, CASCADE или SET NULL. Это означает, что если строка в главной/ссылочной таблице изменяется (обновляется или удаляется), все ссылающиеся таблицы проверяются на допустимость операции и необходимость дополнительных действий; см. документацию PostgreSQL по внешним ключам для подробностей.

Это означает, что при изменении доступа к определённой строке в ссылочной таблице выполняется соответствующее чтение всех ссылающихся таблиц с использованием столбцов внешнего ключа в качестве ключа доступа. Следовательно, все правила проектирования, которые следует учитывать для хорошей производительности чтения, обычно применимы и к столбцам в ссылающейся таблице. Для хорошей производительности в большинстве случаев рекомендуется индекс. PostgreSQL никак этого не требует, поэтому решение остаётся за проектировщиком базы данных.

Для удобства pgAdmin III предоставляет простой флажок при создании внешнего ключа, который автоматически проверит наличие подходящего индекса и создаст его, если таковой не найден. Как и со всеми индексами, бывают редкие случаи, когда такой индекс незначительно снижает общую производительность, но в целом иметь на один индекс меньше гораздо более пагубно для производительности, чем иметь на один индекс больше.
